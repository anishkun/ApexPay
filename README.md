# 🏦 ApexPay: Distributed Event-Driven Financial Ledger

ApexPay is an enterprise-grade backend engine designed to process financial transactions with absolute data integrity. Built with Spring Boot, PostgreSQL, and RabbitMQ, it solves complex distributed system challenges including deadlocks, race conditions, double-spending, and the dual-write problem.

> **Note for Reviewers:** A distributed, event-driven financial ledger built with Spring Boot, PostgreSQL, and RabbitMQ, demonstrating Transactional Outbox, Idempotency, and Pessimistic Locking.

## 🚀 Key Architectural Features

* **Deadlock-Resistant Engine:** Implements lexicographical lock ordering (sorting UUIDs) to mathematically prevent database deadlocks during highly concurrent bi-directional transfers.
* **Double-Spend Prevention:** Utilizes JPA Pessimistic Write Locking (`@Lock(LockModeType.PESSIMISTIC_WRITE)`) to serialize access to account rows, preventing race conditions.
* **Idempotency Facade:** Safely handles network glitches and client retries using an `Idempotency-Key` header. Duplicate requests return cached receipts without re-triggering the ledger.
* **Immutable Audit Trail:** Secures transaction logs at both the application tier (Lombok `@Builder` / no setters) and the database tier (PostgreSQL Triggers blocking `UPDATE` and `DELETE` commands).
* **Transactional Outbox Pattern:** Solves the dual-write problem by saving RabbitMQ events to an internal `outbox_events` table within the same ACID transaction boundary as the financial transfer.
* **Event-Driven Decoupling:** Employs a background relay worker (`@Scheduled`) to reliably publish JSON-serialized success events to RabbitMQ (`TopicExchange`), allowing decoupled microservices (e.g., AI Fraud Detection, Notifications) to consume data without impacting API latency.
* **Migration-Driven Schema (Flyway):** Production runs on PostgreSQL with `ddl-auto=validate` — the schema is owned by versioned **Flyway** migrations, never auto-generated by Hibernate. Hibernate only validates that the entity mappings match the migrated schema.
* **Resilient Outbox Relay:** Publisher **confirms** mean an event is marked `PUBLISHED` only after the broker ACKs it (no loss); failed publishes retry with **bounded attempts + exponential backoff** and exhausted events are dead-lettered. Row-level `FOR UPDATE SKIP LOCKED` + optimistic `@Version` locking guarantee no double-publish.
* **Dead-Letter Queues (DLQ):** Two DLQs isolate failures — one for consumer-side poison messages, one for outbox events that exhaust publish retries — so nothing is silently dropped or infinitely requeued.

## ✅ Build & Test

The project uses a **two-tier testing model** so day-to-day development needs no Docker, while
the production data path (Flyway-on-Postgres, broker confirms, DLQ routing, concurrency) is
proven end-to-end against real infrastructure in CI/local full runs.

```sh
mvn test          # or: ./mvnw test
```

**Tier 1 — fast unit / slice tests (no Docker).**
Application logic runs against an in-memory **H2** database with `ddl-auto=create-drop` and
**Flyway disabled** (the default test profile). These cover controllers, services, idempotency,
locking, and the relay's pure logic with a mocked broker — they run anywhere with no daemon.

**Tier 2 — Testcontainers integration tests (Docker required).**
The integration tests spin up throwaway **PostgreSQL 16** and **RabbitMQ** containers and exercise
the real pipeline: Flyway owns the schema, Hibernate runs with `ddl-auto=validate`, and the relay
publishes through a live broker. A dedicated `pg` Spring profile
(`src/test/resources/application-pg.properties`, selected via `@ActiveProfiles("pg")`) flips the
test datasource from H2 to the container Postgres. The containerized ITs are:

| Integration test | What it proves |
| ---------------- | -------------- |
| `FlywayMigrationIntegrationTest` | `V1` + `V2` migrations apply cleanly and `ddl-auto=validate` passes (entities match the migrated schema) |
| `IdempotencyConcurrencyPostgresIntegrationTest` | 24 concurrent transfers on one `Idempotency-Key` produce **exactly one** debit (no double-spend) |
| `OutboxRabbitIntegrationTest` | Full outbox → exchange → queue delivery with publisher confirms |
| `ConsumerDlqRoutingIntegrationTest` | Poison messages exhaust listener retries and are dead-lettered to the consumer DLQ |
| `OutboxPublishFailureDlqIntegrationTest` | Events that exhaust publish retries become `DEAD` and route to the publish-failure DLQ |

> **Docker is required to run the FULL suite.** When no Docker daemon is reachable, the
> Testcontainers ITs **self-skip cleanly** (the rest of the suite still runs green), so
> `mvn test` never goes red purely because Docker is absent. With Docker present the whole
> suite (35 tests) runs and the ITs above all execute.

Build a runnable jar with `mvn clean package` (jar lands in `target/`).

## 💻 Local Setup & Running

> Running the live application (unlike the tests) requires Postgres and RabbitMQ.

1. Start Infrastructure (Docker)

The quickest way is the bundled compose file:

```sh
docker compose up -d
```

This starts **PostgreSQL 16** (`ApexDB`, user `postgres` / `password`, on `localhost:5432`) and
**RabbitMQ** with the management UI at http://localhost:15672 (guest/guest). Equivalent manual
commands:

```sh
docker run -d --name apexpay-postgres -e POSTGRES_DB=ApexDB -e POSTGRES_USER=postgres -e POSTGRES_PASSWORD=password -p 5432:5432 postgres:16
docker run -d --name apexpay-rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3.13-management
```

**Database connection & credentials.** The app connects to
`jdbc:postgresql://localhost:5432/ApexDB`. Credentials are read from environment variables with
local defaults:

| Env var | Default | Used for |
| ------- | ------- | -------- |
| `DB_USERNAME` | `postgres` | Postgres datasource user |
| `DB_PASSWORD` | `password` | Postgres datasource password |

On first boot **Flyway runs the migrations** in `src/main/resources/db/migration/` and Hibernate
then validates its entity mappings against that schema (`ddl-auto=validate`). The schema is fully
migration-driven — the app never auto-generates or alters tables.

2. Configure Database Triggers (optional, for tamper-proofing)
For a production-grade immutable audit trail, access the PostgreSQL instance and apply the
trigger below. The application also enforces immutability at the code tier (no setters), so
this step is optional for a local run:

```text
CREATE OR REPLACE FUNCTION prevent_audit_log_modification() RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION 'Modifying or deleting records in the audit_logs table is strictly prohibited.';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_append_only_audit_logs
BEFORE UPDATE OR DELETE ON audit_logs
FOR EACH ROW EXECUTE FUNCTION prevent_audit_log_modification();
```
3. Run the Application

```sh
mvn spring-boot:run
```

The app exposes endpoints on `localhost:8080`. The `OutboxRelayWorker` polls the outbox table
every 5 seconds and publishes pending events to RabbitMQ; `RabbitMQConsumer` logs anything that
lands on the AI queue.

## 🗄️ Schema Migrations (Flyway)

Production runs on PostgreSQL with the schema **owned by Flyway**, not Hibernate:

```properties
spring.jpa.hibernate.ddl-auto=validate   # Hibernate validates, never alters
spring.flyway.enabled=true               # Flyway owns/applies the schema
```

Versioned migrations live in `src/main/resources/db/migration/`:

| Version | File | Contents |
| ------- | ---- | -------- |
| `V1` | `V1__baseline_schema.sql` | The 5 baseline tables: `accounts`, `transactions`, `outbox_events`, `idempotency_records`, `audit_logs`. Authored to match exactly what the JPA entities map to (snake_case physical naming, Postgres dialect) so `validate` passes. |
| `V2` | `V2__outbox_resilience.sql` | Adds the resilient-relay columns to `outbox_events` (`status`, `attempt_count`, `next_attempt_at`, `last_error`, `published_at`, `version`), the relay poll index `idx_outbox_events_relay (status, next_attempt_at, created_at)`, and backfills already-relayed rows (`processed = true` → `status = 'PUBLISHED'`). |

Flyway support is provided by `flyway-core` plus `flyway-database-postgresql` (the latter is
required for Postgres on modern Flyway).

**Adding a new migration:** create `V<n>__<description>.sql` in the migration folder with the next
version number (e.g. `V3__add_foo.sql`) and keep it **forward-only** and consistent with the JPA
entities — because prod runs `ddl-auto=validate`, any entity change must ship with a migration or
boot fails. The `FlywayMigrationIntegrationTest` IT verifies migrations apply and validation passes
against a real Postgres container.

## 📨 Resilient Outbox Relay

The business write (transfer + outbox row + idempotency record) commits **atomically in one
`@Transactional`** with **no broker I/O inside the transaction**. A background relay
(`service/OutboxRelayService` + `worker/OutboxRelayWorker`) publishes events **after commit**, so
the database and broker can never disagree (the dual-write problem is solved by the outbox).

**No loss (publisher confirms).** Confirms are enabled globally
(`spring.rabbitmq.publisher-confirm-type=correlated`); the relay blocks on
`waitForConfirms(...)` and only marks an event `PUBLISHED` **after the broker ACKs it**. An
un-ACKed event stays eligible and is retried.

**No double-publish.** Each relay poll locks its batch with Postgres `FOR UPDATE SKIP LOCKED` so
concurrent relays grab disjoint rows, and an optimistic `@Version` column guards each row. The
RabbitMQ message id is the **outbox event UUID**, giving consumers a stable key for dedupe under
**at-least-once** delivery.

**Bounded retry + exponential backoff.** A failed publish reschedules the event with
`delay(attempt) = min(backoff-base-ms × 2^(attempt-1), backoff-max-ms)`. After `max-attempts` the
event is marked `DEAD` and routed to the publish-failure DLQ (see below).

Tunables (bound from `apexpay.outbox.*` via `OutboxProperties`, defaults in
`application.properties`):

| Property | Default | Meaning |
| -------- | ------- | ------- |
| `apexpay.outbox.max-attempts` | `5` | Publish attempts before an event becomes `DEAD` |
| `apexpay.outbox.backoff-base-ms` | `1000` | Base backoff delay (after the first failure) |
| `apexpay.outbox.backoff-max-ms` | `60000` | Cap on the exponential backoff delay |
| `apexpay.outbox.batch-size` | `100` | Max events fetched per relay poll |
| `apexpay.outbox.confirm-timeout-ms` | `5000` | How long to wait for a broker confirm before treating it as a failure |
| `apexpay.outbox.use-skip-locked` | `true` (prod/pg) | Use `FOR UPDATE SKIP LOCKED` poll (`false` on H2, which lacks it) |

## ☠️ Dead-Letter Queues (DLQ)

Two independent DLQ paths isolate the two ways a message can fail, so nothing is dropped or
infinitely requeued (topology declared in `config/RabbitMQConfig`):

**1. Consumer-side poison messages.** The main topic exchange `apexpay.events` routes to the
durable main queue `apexpay.ai.agent.queue`, which carries `x-dead-letter-exchange` args pointing
at the consumer DLX `apexpay.events.dlx` → DLQ `apexpay.ai.agent.queue.dlq` (routing key
`ai.agent.dead`). The `@RabbitListener` container does **bounded retry (3 attempts, exponential
backoff 1s→10s)** with `defaultRequeueRejected=false` and a `RejectAndDontRequeueRecoverer`, so a
message that keeps failing is **dead-lettered after exhaustion** instead of requeued forever.

**2. Publish-failure (outbox).** Outbox events that exhaust their publish retries (status `DEAD`)
are routed by the relay to the publish-failure DLX `apexpay.outbox.dlx` → DLQ `apexpay.outbox.dlq`
(routing key `outbox.publish.failed`) for ops to inspect/replay.

| Path | Exchange | Queue | Routing key |
| ---- | -------- | ----- | ----------- |
| Main delivery | `apexpay.events` (topic) | `apexpay.ai.agent.queue` | `transaction.success.#` |
| Consumer DLQ | `apexpay.events.dlx` (direct) | `apexpay.ai.agent.queue.dlq` | `ai.agent.dead` |
| Publish-failure DLQ | `apexpay.outbox.dlx` (direct) | `apexpay.outbox.dlq` | `outbox.publish.failed` |

> **⚠️ Rollout / ops note.** The main queue `apexpay.ai.agent.queue` is durable and now declares
> the `x-dead-letter-*` arguments. Re-declaring it on a **live broker that already has the old
> queue** (declared without those args) will fail with a `PRECONDITION_FAILED` conflict — RabbitMQ
> won't mutate an existing queue's arguments. A rollout therefore needs a **queue rename or a
> drain-and-delete migration** of the old queue. This is a non-issue in tests because each run uses
> a fresh Testcontainers broker.
